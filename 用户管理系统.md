## v1.0 → v2.0 安全修复详表

### 🔴 严重漏洞（P0）

---

#### 漏洞 01：密码明文存储

| 维度 | 说明 |
|------|------|
| **风险等级** | 🔴 严重 |
| **涉及文件** | `app.py` — USERS 字典定义 |
| **攻击场景** | 服务器被入侵 → 读取源码 → 所有密码直接暴露 |

```diff
# v1.0（漏洞代码）
- USERS = {
-     "admin": {
-         "password": "admin123",          # ← 明文！任何人都能直接读取
-     }
- }

# v2.0（修复方案）
+ from werkzeug.security import generate_password_hash
+
+ USERS = {
+     "admin": {
+         "password": generate_password_hash("admin123"),  # ← bcrypt 加盐哈希
+     }
+ }
```

**具体修改操作：**
1. `app.py` 文件开头新增导入：`from werkzeug.security import generate_password_hash, check_password_hash`
2. 将 USERS 字典中所有密码值从字符串字面量改为 `generate_password_hash("xxx")` 调用
3. 新增依赖包：`werkzeug`（写入 `requirements.txt`）

**原理：** `generate_password_hash()` 使用 bcrypt 算法生成加盐哈希（格式为 `$2b$12$...`），每次生成的哈希值不同，即使数据库泄露也无法逆向还原原始密码。

#### 漏洞 02：密码明文比对

| 维度 | 说明 |
|------|------|
| **风险等级** | 🔴 严重 |
| **涉及文件** | `app.py` — login 路由中的密码校验代码 |
| **攻击场景** | 攻击者可利用时序攻击逐字符猜测密码 |

```diff
# v1.0（漏洞代码）
- if USERS[username]["password"] == password:           # ← 直接 == 明文比对
-     session["username"] = username

# v2.0（修复方案）
+ user = USERS.get(username)
+ if user is None or not check_password_hash(user["password"], password):  # ← 哈希比对
+     return render_template("login.html", error="用户名或密码错误")
+ session["username"] = username
```

**具体修改操作：**
1. 将 `login` 路由中的 `==` 直接比对替换为 `check_password_hash()` 函数调用
2. 先通过 `USERS.get(username)` 获取用户对象，若不存在直接返回错误（防止 KeyError 异常泄露信息）

**原理：** `check_password_hash()` 使用**常量时间比较**（constant-time comparison），无论密码前缀是否匹配，耗时相同，从根本上防御时序攻击（Timing Attack）。

#### 漏洞 03：密码明文展示在页面

| 维度 | 说明 |
|------|------|
| **风险等级** | 🔴 严重 |
| **涉及文件** | `templates/index.html` — 用户信息展示模板 |
| **攻击场景** | 任意登录用户 F12 查看页面源码即可看到密码 |

```diff
# v1.0 index.html（漏洞代码）
  <ul class="info-list">
-     <li><span class="label">密码：</span>{{ user.password }}</li>  <!-- ← 密码直接显示 -->
      <li><span class="label">用户名：</span>{{ user.username }}</li>
      <li><span class="label">邮箱：</span>{{ user.email }}</li>

# v2.0 app.py（修复方案 — 新增过滤函数）
+ def get_safe_user_info(username):
+     """返回不包含密码字段的用户信息"""
+     user = USERS.get(username)
+     if user is None:
+         return None
+     return {k: v for k, v in user.items() if k != "password"}  # ← 过滤掉 password

# v2.0 app.py（修复方案 — 渲染时使用安全函数）
+ user_info = get_safe_user_info(username)
+ return render_template("index.html", user=user_info)
```

**具体修改操作：**
1. `app.py` 中新增 `get_safe_user_info()` 工具函数，复制一份用户字典并过滤掉 `password` 键
2. `index` 路由原代码 `return render_template("index.html", user=USERS.get(username))` 改为调用 `get_safe_user_info(username)`
3. `login` 路由登录成功后也改为使用 `get_safe_user_info(username)` 传入模板
4. 删除 `index.html` 中显示密码的 `<li>` 标签

**原理：** 后端在传递数据到模板之前就主动过滤敏感字段，即使前端模板有漏洞也不会泄露密码。

#### 漏洞 04：Secret Key 硬编码

| 维度 | 说明 |
|------|------|
| **风险等级** | 🔴 严重 |
| **涉及文件** | `app.py` — Flask Secret Key 配置 |
| **攻击场景** | 攻击者用公开密钥伪造任意用户的 session cookie → 越权登录 |

```diff
# v1.0（漏洞代码）
- app.secret_key = "dev-key-2025"     # ← 公开已知字符串，在 GitHub 上可搜索到

# v2.0（修复方案）
+ import secrets
+ app.config["SECRET_KEY"] = secrets.token_hex(32)  # ← 每次启动随机生成 256 位密钥
```

**具体修改操作：**
1. `app.py` 文件开头新增导入：`import secrets`
2. 将 `app.secret_key = "dev-key-2025"` 替换为 `app.config["SECRET_KEY"] = secrets.token_hex(32)`
3. 改用 `app.config.update()` 批量配置的方式管理配置项

**攻击复现：**
```bash
# 攻击者已知密钥 "dev-key-2025"，可伪造任意用户 session
flask-unsign --sign --cookie "{'username':'admin'}" --secret "dev-key-2025"

# 生成的伪造 cookie 可直接替换真实 cookie，无需密码即可登录 admin
```

**原理：** Flask 使用 Secret Key 对 session cookie 进行签名。若密钥固定且公开，任何人可伪造任意用户的 session。`secrets.token_hex(32)` 生成 256 位（32 字节）的随机十六进制字符串，不可预测。

#### 漏洞 05：Debug 模式开启

| 维度 | 说明 |
|------|------|
| **风险等级** | 🟠 高危 |
| **涉及文件** | `app.py` — `app.run()` 启动参数 |
| **攻击场景** | 访问 `/__debugger__` 端点可获取 Python 交互式 shell → RCE |

```diff
# v1.0（漏洞代码）
- app.run(debug=True, host="0.0.0.0", port=5000)    # ← debug=True 开启调试器

# v2.0（修复方案）
+ app.run(debug=False, host="127.0.0.1", port=5000)  # ← debug=False 关闭调试器
```

**具体修改操作：** `app.run()` 的 `debug` 参数从 `True` 改为 `False`

**原理：** Flask/Werkzeug 调试器在 debug 模式下会暴露 `/__debugger__` 端点，攻击者可通过该端点执行任意 Python 代码（远程代码执行 RCE）。生产环境必须关闭。

| 维度 | 说明 |
|------|------|
| **风险等级** | 🟠 高危 |
| **涉及文件** | `app.py` — `app.run()` 主机绑定参数 |
| **攻击场景** | 局域网/公网任意主机可访问该服务 |

```diff
# v1.0（漏洞代码）
```

#### 漏洞 07：无暴力破解防护

| 维度 | 说明 |
|------|------|
| **风险等级** | 🟠 高危 |
| **涉及文件** | `app.py` — login 路由及全局配置 |
| **攻击场景** | 攻击者可无限次尝试不同密码组合，最终猜中密码 |

```diff
# v1.0（漏洞代码）
- @app.route("/login", methods=["GET", "POST"])
- def login():                     # ← 无频率限制，每秒可尝试数百次

# v2.0（修复方案 — 新增限流装饰器 + 初始化）
+ from flask_limiter import Limiter
+ from flask_limiter.util import get_remote_address
+
+ limiter = Limiter(
+     app=app,
+     key_func=get_remote_address,          # ← 按 IP 限流
+     default_limits=["200 per day", "60 per hour"],
+     storage_uri="memory://",
+ )
+
+ @app.route("/login", methods=["GET", "POST"])
+ @limiter.limit("10 per minute")           # ← 登录接口每分钟最多 10 次
+ def login():
```

**具体修改操作：**
1. 新增导入：`from flask_limiter import Limiter` 和 `from flask_limiter.util import get_remote_address`
2. 创建全局 `Limiter` 实例，配置默认限制"每天 200 次、每小时 60 次"
3. 在 `login` 路由上添加 `@limiter.limit("10 per minute")` 装饰器
4. 新增依赖包：`flask-limiter`（写入 `requirements.txt`）

**原理：** 限流器基于 IP 地址追踪请求频率，超过限制后自动返回 HTTP 429（Too Many Requests），阻止自动化暴力破解工具（如 Hydra、Burp Suite Intruder）的批量尝试。

#### 漏洞 08：无 CSRF 防护

| 维度 | 说明 |
|------|------|
| **风险等级** | 🟠 高危 |
| **涉及文件** | `app.py` + `templates/login.html` |
| **攻击场景** | 用户访问恶意网站 → 自动向本系统提交登录 POST → session 被劫持 |

```diff
# v1.0 app.py（漏洞代码）
- app = Flask(__name__)            # ← 无 CSRF 保护

# v2.0 app.py（修复方案）
+ from flask_wtf.csrf import CSRFProtect
+
+ app = Flask(__name__)
+ csrf = CSRFProtect(app)          # ← 启用全局 CSRF 防护
```

```diff
# v1.0 login.html（漏洞代码）
  <form method="post" action="/login">
-     <!-- 无 CSRF token，任意站点可构造表单提交 -->
      <input type="text" name="username">
      <input type="password" name="password">
      <button type="submit">登录</button>
  </form>

# v2.0 login.html（修复方案）
  <form method="post" action="/login">
+     <input type="hidden" name="csrf_token" value="{{ csrf_token() }}">  <!-- ← 新增 token -->
      <input type="text" name="username">
      <input type="password" name="password">
      <button type="submit">登录</button>
  </form>
```

**具体修改操作：**
1. `app.py` 新增导入：`from flask_wtf.csrf import CSRFProtect`
2. 实例化 `csrf = CSRFProtect(app)` 全局启用 CSRF 防护
3. `login.html` 表单内新增隐藏输入 `<input type="hidden" name="csrf_token" value="{{ csrf_token() }}">`
4. 新增依赖包：`flask-wtf`（写入 `requirements.txt`）

**原理：** CSRF 保护为每个用户会话生成一次性 token，嵌入表单。服务端处理 POST 请求时验证 token 是否存在且匹配。攻击者的恶意网站无法获取该 token，因此构造的表单提交会被拒绝（HTTP 400）。

#### 漏洞 10：Session 无过期机制

| 维度 | 说明 |
|------|------|
| **风险等级** | 🟡 中危 |
| **涉及文件** | `app.py` — Session 配置 |
| **攻击场景** | 用户使用公共电脑后不登出 → 他人可继续操作该账户 |

```diff
# v1.0（漏洞代码）
- app.config["SECRET_KEY"] = "dev-key-2025"      # ← Flask 默认 31 天超时，未显式配置

# v2.0（修复方案 — 新增 session 配置块）
+ app.config.update(
+     SECRET_KEY=secrets.token_hex(32),
+     SESSION_PERMANENT=False,                    # ← session 随浏览器关闭失效
+     PERMANENT_SESSION_LIFETIME=timedelta(hours=2),  # ← 即使存了持久 cookie，2 小时过期
+     SESSION_COOKIE_HTTPONLY=True,
+     SESSION_COOKIE_SAMESITE="Lax",
+ )
```

**具体修改操作：**
1. 新增导入：`from datetime import timedelta`
2. 新增 `SESSION_PERMANENT = False` — 关闭持久 session（关闭浏览器即失效）
3. 新增 `PERMANENT_SESSION_LIFETIME = timedelta(hours=2)` — 即使开启持久，2 小时强制过期
4. 新增 `SESSION_COOKIE_HTTPONLY = True` — 禁止 JavaScript 读取 cookie（防 XSS 窃取）
5. 新增 `SESSION_COOKIE_SAMESITE = "Lax"` — 限制跨站请求携带 cookie

**原理：** Flask 默认 session 有效期 31 天。通过配置缩短有效期并关闭持久化，降低 session 劫持和会话固定的风险窗口。

#### 漏洞 11：无日志审计

| 维度 | 说明 |
|------|------|
| **风险等级** | 🟡 中危 |
| **涉及文件** | `app.py` — 全局 |
| **攻击场景** | 入侵发生后无日志可查，无法追踪攻击来源 |

```diff
# v1.0（漏洞代码）
- # 无日志配置，无任何操作记录

# v2.0（修复方案 — 新增日志系统）
+ import logging
+
+ logging.basicConfig(
+     level=logging.INFO,
+     format="%(asctime)s [%(levelname)s] %(message)s",
+ )
+ logger = logging.getLogger("access")

# v1.0 login 路由（无日志）
- session["username"] = username         # 登录成功，无记录

# v2.0 login 路由（新增日志埋点）
+ logger.warning(f"Login with empty fields from {request.remote_addr}")
+ logger.info(f"Failed login for '{username}' from {request.remote_addr}")
+ logger.info(f"Successful login for '{username}' from {request.remote_addr}")
```

**具体修改操作：**
1. `app.py` 新增 `import logging` + 配置 logging 格式和级别
2. `login` 路由新增 3 个日志埋点（空字段、登录失败、登录成功），均记录 IP 地址
3. `logout` 路由新增日志：`User '{username}' logged out`

**日志输出示例：**
```
2026-07-19 14:23:10 [INFO] Failed login for 'admin' from 192.168.1.100
2026-07-19 14:23:15 [INFO] Successful login for 'admin' from 127.0.0.1
2026-07-19 14:23:20 [INFO] User 'admin' logged out
```

**原理：** 日志是安全审计的基础。记录登录成功/失败事件及来源 IP，可在攻击发生后追溯时间线和攻击源，也是暴力破解检测的重要数据来源。

#### 漏洞 12：无输入校验

| 维度 | 说明 |
|------|------|
| **风险等级** | 🟡 中危 |
| **涉及文件** | `app.py` — login 路由输入处理 |
| **攻击场景** | 超长输入导致服务端内存耗尽（DoS）或触发底层缓冲区溢出 |

```diff
# v1.0（漏洞代码）
- username = request.form.get("username")   # ← 直接从表单取值，无任何校验
- password = request.form.get("password")

# v2.0（修复方案）
+ username = (request.form.get("username") or "").strip()  # ← 空值兜底 + 去除首尾空格
+ password = request.form.get("password") or ""
+
+ # 输入基本校验
+ if not username or not password:                          # ← 空值检测
+     return render_template("login.html", error="用户名和密码不能为空")
+
+ if len(username) > 64 or len(password) > 128:             # ← 长度限制
+     return render_template("login.html", error="输入内容过长")
```

**具体修改操作：**
1. 对表单取值增加 `or ""` 空值兜底，避免对 `None` 调用 `.strip()` 报错
2. 用户名增加 `.strip()` 去除首尾空格
3. 新增空值检测：用户名或密码为空时返回错误
4. 新增长度限制：用户名不超过 64 字符、密码不超过 128 字符

**原理：** 输入校验是安全编码的第一道防线。空值检测防止逻辑绕过，长度限制防止超长输入导致的资源耗尽（ReDoS / Buffer Overflow），同时减少数据库存储压力。

## v3.0 — SQL 注入漏洞与修复说明

v3.0 在 v2.0 的安全加固基础上，**保持原有登录功能不变**，新增了 SQLite 数据库、用户注册和用户搜索三大功能。初始构建时使用了 **f-string 字符串拼接** 构建 SQL 语句，存在 SQL 注入漏洞，以下是漏洞分析与修复过程。

### v3.0 新增功能概述

| 功能 | 说明 |
|------|------|
| **SQLite 数据库** | `data/users.db`，建 `users` 表（id, username, password, email, phone），插入 admin/alice 默认用户 |
| **用户注册 `/register`** | GET 显示表单，POST 写入数据库 |
| **用户搜索 `/search`** | GET 按用户名/邮箱模糊搜索，结果表格展示 |

### 🔴 漏洞 13：注册功能 SQL 注入

| 维度 | 说明 |
|------|------|

### 🔴 漏洞 14：搜索功能 SQL 注入

| 维度 | 说明 |
|------|------|
| **风险等级** | 🔴 严重 |
| **涉及文件** | `app.py` — search 路由 |
| **攻击场景** | 攻击者在搜索框输入 SQL 构造字符，使 LIKE 子句被绕过，获取全部用户数据甚至通过 UNION 注入窃取其他表信息 |

```diff

- def search():

-             cur.execute(sql)            # ← 直接执行拼接后的 SQL

+ def search():

+     username = session.get("username")
+     if not username:
+         return redirect("/login")
+
+     user_info = get_safe_user_info(username)
+     keyword = (request.args.get("keyword") or "").strip()
+
+     # 关键词长度限制
+     if len(keyword) > 100:
+         keyword = keyword[:100]
+
+     results = []
+     if keyword:
+         conn = sqlite3.connect("data/users.db")
+         conn.row_factory = sqlite3.Row
+         cur = conn.cursor()
+         # ✅ 参数化查询 — LIKE 通配符由参数值提供
+         sql = "SELECT * FROM users WHERE username LIKE ? OR email LIKE ?"
+         like_param = f"%{keyword}%"
+         logger.info(f"[SEARCH] 搜索关键词: '{keyword}'")
+         try:
+             cur.execute(sql, (like_param, like_param))  # ← 参数自动转义
+             rows = cur.fetchall()
```

**具体修改操作：**
1. SQL 语句从 f-string 拼接改为 `?` 占位符参数化查询，`%keyword%` 通配符作为参数值传入
2. 新增登录状态检查：未登录用户直接重定向到 `/login`
3. 新增关键词长度限制（最长 100 字符），超长截断
4. 移除 `print(f"[SEARCH] SQL: {sql}")`（生产环境不暴露 SQL）
5. `logger.info` 改为记录安全的关键词文本而非完整 SQL

**注入原理：**
```
搜索 ?keyword=' OR 1=1 --
拼接结果: SELECT * FROM users WHERE username LIKE '%' OR 1=1 -- %'
WHERE 条件恒为真，返回 users 表中全部数据

搜索 ?keyword=' UNION SELECT 1,'admin','pwd','email','phone' --
拼接结果: ... UNION SELECT ... 窃取其他表数据
```

**修复原理：** 参数化查询将通配符 `%` 作为参数值的一部分传入，而不是拼接到 SQL 模板中。`LIKE ?` 中的参数同样会被 SQLite 驱动安全转义。登录校验确保搜索功能只能被认证用户使用，减少攻击面。
