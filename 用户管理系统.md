## 修复详情

### 漏洞 01：密码明文存储

**问题代码（v1.0）：**
```python
USERS = {
    "admin": {
        "password": "admin123",   # 明文存储
    }
}
```

**修复方案（v2.0）：**
```python
from werkzeug.security import generate_password_hash

USERS = {
    "admin": {
        "password": generate_password_hash("admin123"),  # bcrypt 哈希
    }
}
```

**原理：** `generate_password_hash()` 使用 bcrypt 加盐哈希，即使数据库泄露也无法逆向还原。

---

### 漏洞 02：密码明文比对

**问题代码（v1.0）：**
```python
if USERS[username]["password"] == password:  # 直接 == 比对
```

**修复方案（v2.0）：**
```python
if check_password_hash(user["password"], password):  # 安全哈希比对
```

**原理：** 常量时间比较，防止时序攻击。

---

### 漏洞 03：密码明文展示在页面

**问题代码（v1.0 index.html）：**
```html
<li><span class="label">密码：</span>{{ user.password }}</li>
```

**修复方案（v2.0）：** 新增 `get_safe_user_info()` 函数过滤 password 字段，模板中删除密码行。

---

### 漏洞 04：Secret Key 硬编码

**问题代码（v1.0）：**
```python
app.secret_key = "dev-key-2025"  # 公开已知的固定字符串
```

**修复方案（v2.0）：**
```python
import secrets
app.config["SECRET_KEY"] = secrets.token_hex(32)  # 256 位随机密钥
```

**攻击演示：**
```bash
flask-unsign --sign --cookie "{'username':'admin'}" --secret "dev-key-2025"
```

---

### 漏洞 05：Debug 模式开启

**v1.0：** `app.run(debug=True, host="0.0.0.0", port=5000)`
**v2.0：** `app.run(debug=False, host="127.0.0.1", port=5000)`

Werkzeug 调试器 `/__debugger__` 端点可能导致远程代码执行。

---

### 漏洞 06：绑定 0.0.0.0

**v1.0：** `host="0.0.0.0"` — 所有网络接口可达
**v2.0：** `host="127.0.0.1"` — 仅本机可访问

---

### 漏洞 07：无暴力破解防护

**v1.0：** 登录接口无任何限制
**v2.0：** 集成 `flask-limiter`，每分钟最多 10 次尝试，超限返回 HTTP 429

```python
@limiter.limit("10 per minute")
def login():
    ...
```

---

### 漏洞 08：无 CSRF 防护

**v1.0：** POST 请求无 CSRF token
**v2.0：** 启用 `CSRFProtect(app)`，每个表单嵌入 csrf_token

无 token 的 POST 直接返回 HTTP 400。

---

### 漏洞 09：HTML 注释泄露账号

**v1.0：**
```html
<!-- 调试信息 - 默认管理员账号 用户名: admin 密码: admin123 -->
```
**v2.0：** 已删除，替换为安全通用提示。

---

### 漏洞 10：Session 无过期机制

**v1.0：** 未配置超时，默认 31 天
**v2.0：** `SESSION_PERMANENT=False`，`PERMANENT_SESSION_LIFETIME=2 小时`

---

### 漏洞 11：无日志审计

**v1.0：** 无任何日志
**v2.0：** 集成 logging，记录登录成功/失败/登出操作及 IP 地址

```
2026-07-19 [INFO] Failed login for 'admin' from 192.168.1.100
2026-07-19 [INFO] Successful login for 'admin' from 127.0.0.1
```

---

### 漏洞 12：无输入校验

**v1.0：** `request.form.get()` 直接使用
**v2.0：** 空值检测 + 用户名 64 字符、密码 128 字符长度限制

## v3.0 新增功能与教学漏洞说明

v3.0 在 v2.0 的安全加固基础上，**保持原有登录功能不变**，新增了以下三个核心变更：

### 变更一：引入 SQLite 数据库

| 项目 | 说明 |
|------|------|
| **动机** | v2.0 用户数据全部写死在内存字典中，无法持久化，也无法演示数据库相关攻击 |
| **实现** | 新增 `init_db()` 函数，启动时在 `data/users.db` 创建 `users` 表 |
| **表结构** | `id`(自增主键)、`username`(唯一)、`password`、`email`、`phone` |
| **默认数据** | `admin/admin123`、`alice/alice2025`（`INSERT OR IGNORE` 防重复） |

```python
# v3.0 新增 — database/init.py
def init_db():
    os.makedirs("data", exist_ok=True)
    conn = sqlite3.connect("data/users.db")
    cur = conn.cursor()
    cur.execute("""CREATE TABLE IF NOT EXISTS users (
        id INTEGER PRIMARY KEY AUTOINCREMENT,
        username TEXT UNIQUE NOT NULL,
        password TEXT NOT NULL,
        email TEXT,
        phone TEXT
    )""")
    # 插入 admin 和 alice
    conn.commit()
```

### 变更二：新增用户注册 `/register`

| 项目 | 说明 |
|------|------|
| **路由** | `GET/POST /register` |
| **GET** | 显示 `register.html` 注册表单（用户名、密码、邮箱、手机号） |
| **POST** | 将表单数据通过 **f-string 字符串拼接** 写入 SQLite |
| **成功** | 重定向至 `/login?msg=注册成功，请登录` |
| **失败** | 重复用户名提示"用户名已存在，请换一个" |

**涉及文件：**
- 新增 `app.py` — `/register` 路由
- 新增 `templates/register.html` — 注册页面，继承 base.html

### 变更三：新增用户搜索 `/search`

| 项目 | 说明 |
|------|------|
| **路由** | `GET /search?keyword=` |
| **SQL 逻辑** | `WHERE username LIKE '%keyword%' OR email LIKE '%keyword%'` |
| **SQL 构建** | **f-string 字符串拼接**（非参数化查询） |
| **控制台输出** | `print(f"[SEARCH] SQL: {sql}")` — 打印原始 SQL 便于观察注入 |
| **结果展示** | 首页表格显示 ID、用户名、邮箱、手机 |
| **空结果** | 显示"无搜索结果" |

**涉及文件：**
- 新增 `app.py` — `/search` 路由
- 修改 `templates/index.html` — 增加搜索框和结果表格
- 修改 `templates/base.html` — 未登录导航栏增加"注册"链接

### 🔴 v3.0 新增教学漏洞：SQL 注入（2 处）

不同于 v1.0 的无心之过，v3.0 **刻意引入 SQL 注入漏洞**用于教学演示，方便学生对比"安全代码"与"不安全代码"的区别。

| 编号 | 漏洞描述 | 位置 | 严重程度 |
|:----:|---------|------|:--------:|
| 13 | **注册功能 SQL 注入** — f-string 拼接 INSERT，用户输入直接拼入 SQL | `app.py` register 路由 | 🔴 严重 |
| 14 | **搜索功能 SQL 注入** — f-string 拼接 LIKE 子句，控制台打印原始 SQL | `app.py` search 路由 | 🔴 严重 |

#### 漏洞 13：注册 SQL 注入

**代码（v3.0 app.py）：**
```python
# f-string 直接拼接，无任何过滤或转义
sql = f"INSERT INTO users (username, password, email, phone) VALUES ('{username}', '{password}', '{email}', '{phone}')"
logger.info(f"[REGISTER] 执行 SQL: {sql}")
```

**攻击示例：**
```
用户名输入:  admin', 'hacked'), ('newadmin
→ 拼接结果: INSERT INTO users (...) VALUES ('admin', 'hacked'), ('newadmin', '...', '...', '...')
→ 可插入任意数据，甚至修改表结构
```

#### 漏洞 14：搜索 SQL 注入

**代码（v3.0 app.py）：**
```python
# f-string 直接拼接，控制台打印 SQL
sql = f"SELECT * FROM users WHERE username LIKE '%{keyword}%' OR email LIKE '%{keyword}%'"
print(f"[SEARCH] SQL: {sql}")  # 控制台可见注入效果
```

**攻击示例：**
```
搜索关键词:  ' OR 1=1 --
→ 拼接结果: SELECT * FROM users WHERE username LIKE '%' OR 1=1 -- %'
→ 返回全部用户数据，突破权限限制

搜索关键词:  ' UNION SELECT 1, 'admin', '泄露的密码', 'email', 'phone' --
→ 拼接结果: ... UNION SELECT ...
→ 可窃取其他表的敏感信息
```

> 💡 **教学建议**：在搜索框尝试输入 `' OR 1=1 --` 观察全部用户数据泄露，再对比参数化查询的防御效果。

### v3.0 与 v2.0 关键差异速览

| 对比项 | v2.0 | v3.0 |
|--------|------|------|
| 数据存储 | 内存字典 `USERS = {...}` | 内存字典 + SQLite `data/users.db` |
| 用户注册 | ❌ 无 | ✅ `/register` 表单注册写入数据库 |
| 用户搜索 | ❌ 无 | ✅ `/search?keyword=` 模糊搜索表格展示 |
| 数据库 | ❌ 无 | ✅ SQLite + `init_db()` 自动建表 |
| 注册页面 | ❌ 无 | ✅ `register.html` |
| 搜索功能 | ❌ 无 | ✅ index.html 搜索框 + 结果表格 |
| 导航栏 | 未登录仅显示"登录" | 未登录增加"注册"链接 |
| 登录校验 | USERS 字典 + bcrypt | **不变**（仍用 USERS 字典） |
| SQL 注入教学 | ❌ 无 | ✅ 2 处 f-string 拼接 + 控制台打印 SQL |

> ⚠️ **注意**：v3.0 注册数据写入 SQLite，但登录仍用内存字典 `USERS`，故新注册用户**暂无法登录**。这是"保持原有登录功能不变"的设计约束。
